BAQIS Quafu Skill · Group Meeting Brief

把 Quafu 能力装进用户自己的 AI Agent

这不是一个新的聊天网页，而是一套可安装到 Claude、Codex、opencode、Cursor 等工具里的本地 Agent Skill。它让 agent 在遇到 Quafu / QuarkStudio / QSteed 任务时自动加载正确流程，能配置、写代码、验证，并在真实硬件提交前加安全门。

一句话：baqis-quafu 是面向 Quafu-SQC / QuarkStudio / QSteed 的跨客户端工作能力包。用户安装后，自己的 coding agent 就会在合适时机触发专门知识和脚本，而不是每次重新问“怎么配置”。

谁需要安装？

量子云用户

想用 Quafu-SQC / QuarkStudio 跑量子云任务，需要 agent 协助写 OpenQASM、构造任务、读取结果。

开发者 / 教学者

希望 Claude、Codex、opencode、Cursor 都遵守同一套 Quafu 操作规范，而不是每个工具各写一份提示词。

安全敏感场景

不希望 API token 被打印、写进源码、进入 LLM 上下文，且真实硬件提交必须先确认。

QSteed 使用者

想在本地用 QSteed 做线路编译 / transpile，并把结果接到 Quafu 工作流里。

安装后能提供什么能力？

1

自动触发 Skill

当用户提到 Quafu、QuarkStudio、QSteed、OpenQASM、backend status、token 配置、任务结果等关键词时，agent 自动加载这套能力包。

2

从注册到 token 的安全流程

先确认用户是否已注册 Quafu-SQC；token 只通过 helper 保存到用户本地配置，不复述、不打印、不写入源码。

3

写可运行代码，而不是只给说明

生成 QuarkStudio / OpenQASM 任务代码，检查 SDK import，读取 backend status，并为 result retrieval 生成可复用代码。

4

真实硬件提交安全门

调用 tmgr.run(task) 前，必须明确 backend、circuit、shots、options，并得到用户对这一次提交的当前授权。

5

可选 QSteed 能力 Opt-in

用户同意后才创建隔离 Python venv，安装已验证版本，运行本地 transpile demo；不碰 token、不联网、不提交硬件。

Skill 和“网页里的云端 Agent”有什么区别？

核心区别：网页云端 Agent 是“用户去访问一个服务”；Agent Skill 是“能力被安装进用户自己的工具”。前者集中托管，后者分发知识、流程和脚本。

本地 Agent Skill

  • 安装到用户已有的 Claude / Codex / Cursor / opencode。
  • 运行在用户自己的开发环境里，能读写项目文件、运行本地命令。
  • 适合“帮我配置、写代码、跑测试、生成任务脚本”。
  • 凭证留在用户本地，skill 规定如何保存和避免泄漏。
  • 能力可组合：同一个 skill 可被不同 agent 客户端复用。

网页里的云端 Agent

  • 用户访问一个固定网站或聊天入口。
  • 后端统一托管模型、工具、会话和权限。
  • 适合“平台内问答、在线任务查询、统一产品入口”。
  • 凭证和会话通常由云端服务管理，需要服务端安全架构。
  • 体验统一，但对用户本地项目和开发环境的操作能力较弱。

我们在 QSteed 部分做了什么？

QSteed 能力是可选启用的，不会在安装 skill 时默认改用户环境。启用后，agent 会先展示 dry run，再经用户同意创建隔离 venv，安装本机验证过的版本：

pyquafu==0.4.1
qsteed==0.2.2
python helpers/qsteed_smoke.py --check-import --transpile-demo

这个 smoke check 只做本地 import 和小规模 transpile 验证：不读取 Quafu token、不联网、不提交真实硬件任务。

和 quafu.top 的关系与区别

基于此前在 shared-agent / Claude Code 对话中的调研，quafu.top 更像 BAQIS 自研的在线窄域量子助手：有 Web 前端、FastAPI/Starlette 后端、SSE chat、意图路由和工具调用。

baqis-quafu skill 不是托管服务，也不提供固定网站入口。它是安装到用户本地 agent 的能力包，让用户在自己的开发环境里安全地配置、写代码、验证和调用 Quafu / QSteed。

组会 takeaway：如果目标是“给所有人一个统一入口”，做云端 Agent；如果目标是“让每个开发者自己的 agent 都会正确使用 Quafu / QSteed”，做 Skill。两者不冲突：云端 Agent 是产品入口，Skill 是开发者工作流能力包。
